In addition, there will be a new episode ofSharp Techlater today.On to the Update:The XZ BackdoorFromArs Technica:Researchers have found a malicious backdoor in a compression tool that made its way into widely used Linux distributions, including those from Red Hat and Debian. The compression utility, known asxz Utils, introduced the malicious code in versions 5.6.0 and 5.6.1,according toAndres Freund, the developer who discovered it. There are no known reports of those versions being incorporated into any production releases for major Linux distributions, but bothRed HatandDebianreported that recently published beta releases used at least one of the backdoored versions—specifically, in Fedora Rawhide and Debian testing, unstable and experimental distributions. A stable release of Arch Linux is also affected. That distribution, however, isn’t used in production systems.Because the backdoor was discovered before the malicious versions of xz Utils were added to production versions of Linux, “it’s not really affecting anyone in the real world,” Will Dormann, a senior vulnerability analyst at security firm Analygence, said in an online interview. “BUT that’s only because it was discovered early due to bad actor sloppiness. Had it not been discovered, it would have been catastrophic to the world.”This is a pretty incredible story, and a pretty impressive exploit — with an exceptional hero at the end of it. To summarize (primary sourcehere):xzis a compression tool that generally comes standard in most Linux distributions (thinkzip, but more efficient and better suited to single large files).xzuses LZMA (Lempel-Ziv-Markov chain Algorithm) for compression; LZMA is delivered in a library calledliblzma.systemdis an init system and system manager for Linux. This is the process that both starts the computer and manages system processes once the computer is running.systemd‘s journaling needs to be compressed over time, so it is often packaged withliblzma.OpenSSH is a suite of utilities that manages remote logins to a system via the command line. It is packaged in nearly every Linux/Unix distribution. OpenSSH is a standalone program, but it is often patched to supportsystemdso that other services can be initiated whensshdis running.Note the order here: patched versions of OpenSSH have a dependency onsystemd;systemdhas a dependency onliblzma. That means that when a Linux system is compiled it pulls downliblzmafrom its GitHub site to incorporate it intosystemdand thus patched versions of OpenSSH. Critically, though, it doesn’t pull down the actual source code; rather, it pulls down “tarballs” that are created for various configurations to make the process easier.In this case the tarball forliblzmacontained a payload that was injected into the compiled source for systems that met the hacker’s criteria (x86 processors and Debian or Red Hat packages, which are upstream of most Linux systems, including Ubuntu). The target of this payload wassshd, which was modified to give a specific key root access to the system; if any other key was presented tosshd, it would operate as normal.This is exceptionally clever: basically a special lock was inserted into the remote login system that could only be unlocked — or even exposed — by one key. Indeed, because the inserted vulnerability is only activated when presented by that key, it can’t be discovered by scanning public computers; had this attack succeeded, with the spoiled version ofliblzmadeployed broadly, the majority of the world’s computers would be vulnerable and no one would know.What HappenedSo on to the story of how this happened (most of this summary is courtesy ofEvan Boehs).First off, OpenSSH andsystemddo not have any relevant vulnerabilities here; the problem is withliblzmaspecifically, and thexzproject generally.xzwas created and maintained by a developer namedLasse Collin;xzhas been stable since 2010, with occasional updates since then.In 2021 a certain “Jia Tan” created a GitHub account (@JiaT75, since disabled).In 2022 Jia Tan submitted a patch forxz; that patch, and several other commits to the project itself, were strongly supported by a number of users on the mailing list who, it appears, did not exist anywhere else on the Internet.A few months later, in response to some of these mystery users complaining about Collin’s slowness in incorporating these updates, Collinresponded:I haven’t lost interest but my ability to care has been fairly limited mostly due to longterm mental health issues but also due to some other things. Recently I’ve worked off-list a bit with Jia Tan on XZ Utils and perhaps he will have a bigger role in the future, we’ll see.It’s also good to keep in mind that this is an unpaid hobby project.Anyway, I assure you that I know far too well about the problem that not much progress has been made. The thought of finding new maintainers has existed for a long time too as the current situation is obviously bad and sad for the project.A new XZ Utils stable branch should get released this year with threaded decoder etc. and a few alpha/beta releases before that. Perhaps the moment after the 5.4.0 release would be a convenient moment to make changes in the list of project maintainer(s).Over the next few months the mystery users disappeared, while Jia Tan became a project maintainer, with full commit capabilities; various changes were committed to not justxz, but also submitted to various open source testing suites that might have uncovered the impending vulnerabilities (sometimes via other mystery users with similarly-formatted emails to the mystery users on thexzmailing list). Then, earlier this year, Jia Tan changed the official URL of the project away from Collin’s server to GitHub pages, and uploaded the files that inserted the vulnerability.This is, needless to say, a textbook example of open source’s vulnerabilities. First there is the role of Collin’s himself, as memorialized by xkcd:Collin’s work withxzwas, to be fair, mostly finished; still, software is never truly finished, which meant that his creation and the reward of overwhelming adoption came with a heavy burden: bug-fixing and support for life, and harassment on the mailing list if he wasn’t fast enough. That said harassment seemed to accompany a potential life-saver — Jia Tan — is only obvious in retrospect. Jia Tan solved Collin’s problem, and thus gained access to a part of most Linux distributions that was both unquestioned and also upstream from OpenSSH (but only the patched version), which made this entire attack possible.Second is the reliance on dependencies: there are legitimate reasons to patch OpenSSH, but that meanssshdis dependent onsystemd, which itself is dependent onliblzma, again for very good reasons. Each of those dependencies, though, particularly when chained together, dramatically increase the attack surface for critical code.And so we come to the last part of this story, which is just as incredible as the exploit itself, and how it was inserted in the first place. Last Friday Andres Freund, a Microsoft developer working on the open-source Postgres database,posted to an open source security mailing list:After observing a few odd symptoms aroundliblzma(part of thexzpackage) on Debian sid installations over the last weeks (logins withsshdtaking a lot of CPU,valgrinderrors) I figured out the answer:  The upstreamxzrepository and thexztarballs have been backdoored.  At first I thought this was a compromise of debian’s package, but it turns out to be upstream.The slowdown appears to be connected to the dependency chain I laid out above:liblzmais loaded into memory relatively early, then the nefarious code has to wait forsshdto be loaded and the relevant parts altered to point to the backdoor. Freund, incredibly enough, consistently perceived this slowness, which amounted to about an extra half a second, and decided to investigate.This, it must again be noted, is a textbook example of open source’s strength: yes, Freund was some combination of attentive, skilled, and curious, which was fortunate for not just everyone in tech but, by extension, everyone in the world, but he was also one of many people all over the world using the altered code. The best way to get a “lucky” roll of the dice is to roll the dice as many times as possible.Since then the entire community has been on a mission to uncover everything there is uncover about Jia Tan and all of the oddities surrounding his activity, including all of the apparent sock puppets; this too is a strength of open source: everything from code to commits to emails is out in the open for anyone to scour. Yes, this openness let Jia Tan in; it also found out everything there is to know about him.Open Source SafetyFirst off, no one knows who Jia Tan is. The name is obviously Chinese, but that doesn’t mean anything; what matters more is the level of sophistication and sheer amount of resources, particularly in terms of time, put into the project. That, combined with the scale of the vulnerability — “Jia Tan” nearly had untraceable root level access to a huge number of servers all over the world — strongly points to nation-state involvement (and again, the name is meaningless here, given it is probably made up).Secondly, I do think this ultimately is a win for open source: yes, open source created the entry point for Jia Tan, but it also found the vulnerability; there could very well be a “Jia Tan” working internally on a closed source project who would be less likely to be discovered in a similar way (to use a vulnerability of similar scale, something like iPhone root access at Apple).What is clear is that the community as a whole needs to think more deeply about precisely this sort of attack.This thread by @thegrugqabout how those sketchy accounts created the crisis that Jia Tan could solve is a must-read;this tweet near the end, though, gives a plan of action:Briefly, I want to address the issue of who is to blame. Easy — the people behind the attack. Lasse, the maintainer of xz, was the target of a patient intelligence campaign that invested more resources into subverting him than anyone invested into his project.— thaddeus e. grugq thegrugq@infosec.exchange (@thegrugq)March 31, 2024Open source undergirds nearly everything on the Internet, and for good reason: it simply makes economic sense to share resources for essential software that provides no meaningful differentiation. To that end, big companies in tech contribute a lot to open source; ultimately, though, their incentive is towards growth, which means most of those resources are directed towards building out new features, or maintaining projects that are essential to their business. What no one is directly incentivized to do is check up on the boring but essential blocks likexz, leaving them vulnerable to campaigns like this one.To that end, while I don’t know exactly what this structure would look like, it seems like it does behoove the community to maintain some sort of inventory of essential pieces of code likexz, and for the largest companies to commit resources to, at a minimum, keeping an eye on their commits. Yes, this will be unwieldy, but I doubt this will be the only attack of this sort; indeed, the real task is seeing if similar attacks have already happened.This Update will be available as a podcast later today.
