OpenAI API Updates
------------------

From OpenAI ( https://openai.com/index/new-tools-for-building-agents/ ) :

Today, we’re releasing the first set of building blocks that will help developers and enterprises build useful and reliable agents. We view agents as systems that independently accomplish tasks on behalf of users. Over the past year, we’ve introduced new model capabilities — such as advanced reasoning, multimodal interactions, and new safety techniques — that have laid the foundation for our models to handle the complex, multi-step tasks required to build agents. However, customers have shared that turning these capabilities into production-ready agents can be challenging, often requiring extensive prompt iteration and custom orchestration logic without sufficient visibility or built-in support.

To address these challenges, we’re launching a new set of APIs and tools specifically designed to simplify the development of agentic applications:

* The new Responses API ( https://platform.openai.com/docs/quickstart?api-mode=responses ) , combining the simplicity of the Chat Completions API with the tool use capabilities of the Assistants API for building agents
* Built-in tools including web search ( https://platform.openai.com/docs/guides/tools-web-search ) , file search ( https://platform.openai.com/docs/guides/tools-file-search ) , and computer use ( https://platform.openai.com/docs/guides/tools-computer-use )
* The new Agents SDK ( https://platform.openai.com/docs/guides/agents ) to orchestrate single-agent and multi-agent workflows
* Integrated observability tools ( https://platform.openai.com/docs/guides/agents#orchestration ) to trace and inspect agent workflow execution

These new tools streamline core agent logic, orchestration, and interactions, making it significantly easier for developers to get started with building agents. Over the coming weeks and months, we plan to release additional tools and capabilities to further simplify and accelerate building agentic applications on our platform…

The Responses API is designed for developers who want to easily combine OpenAI models and built-in tools into their apps, without the complexity of integrating multiple APIs or external vendors. The API also makes it easier to store data on OpenAI so developers can evaluate agent performance using features such as tracing and evaluations. As a reminder, we do not train our models on business data by default, even when the data is stored on OpenAI. The API is available to all developers starting today and is not charged separately—tokens and tools are billed at standard rates specified on our pricing page⁠. Check out the Responses API quickstart guide⁠ to learn more.

I think this API release is pretty interesting for a host of reasons.

First, from a developer perspective, this solves one of the biggest pain points from the Chat Completions API ( https://platform.openai.com/docs/api-reference/chat ) : that API, which has become something of an industry standard, was stateless, which meant that developers not only needed to maintain the state of a current conversation, but they also had to pass an increasingly large amount of context back-and-forth to OpenAI to accomplish any multi-step task, whether initiated by the app or the user. The Responses API, on the other hand, maintains state on OpenAI's servers, dramatically simplifying things for developers.

The new tools are interesting as well:

* Web search already existed in the Chat Completions API, but if you wanted to use it you were committed to use it — a web search would be done before a response was generated no matter what; with the Responses API search can be invoked on an as-needed basis.
* File search operates similarly: once you have uploaded the files you want searched to OpenAI's servers, the Responses API will leverage them on an as-needed basis.
* Computer use gives developers the capabilities of Operator, OpenAI's browser-using agent. Unlike file search, however, computer use is just an API; OpenAI isn't hosting any virtual machines for you, so you'll have to do that yourself.

Despite its limitations, the computer use tool is notable because it is delivering the capability that was at the core of a ChatGPT Pro feature that OpenAI released a mere seven weeks ago ( https://openai.com/index/introducing-operator/ ).

Steelmanning the API Business
-----------------------------

This raises the obvious question as what is OpenAI's priority: its products or its API? This exact question came up on the AMA OpenAI conducted on X in the wake of the release:

( https://x.com/kevinweil/status/1899529369430749567 )

Kevin Weil is the Chief Product Officer of OpenAI, and was a former head of product for Instagram. Instagram is notable for the extremity with which it embraced being a product; the only APIs that exist now are for businesses and creators (even the basic Display API, which is all that was left of the old public API that was shut down in 2018, was itself shut down last year ( https://developers.facebook.com/blog/post/2024/09/04/update-on-instagram-basic-display-api/ ) ). Heck, even users are limited to a single link in bio!

This is an entirely rational approach for Instagram: they are an ad-supported business that makes more money the more time more users spend in their apps. An API works against both goals: it gives some number of users a reason to not be on Instagram, and it gives users on Instagram a way to leave. Given that Instagram's addressable market is the entire world (which is a plausible goal given it is fully ad-supported), there is no conceivable way that an API — even if Instagram charged for it — would ever be a profitable endeavor, and that's even before you get to the cost of maintaining it.

OpenAI, on the other hand, is not an ad-supported business, at least not yet. Their product — ChatGPT — monetizes through subscriptions; its API monetizes through usage. These are by-and-large compatible business models, particularly in the overall structure of OpenAI's business.

* OpenAI has three major cost drivers. First, there are its corporate costs, particularly R&D. Second, there are the costs of training; while I don't know how OpenAI accounts for these costs, to my mind they are a capital cost that ought to show up as depreciation in cost-of-goods sold. Third, there are the marginal costs of inference for its users and developers.
* Very large R&D and capital costs argue for maximizing the number of users, so that those costs can be spread out across more users.
* Working in the opposite direction is the fact that marginal costs are a floor for usage; while large fixed costs call for more users, the revenue from every additional user needs to at least cover the marginal cost of serving them.

Both subscriptions and usage charges solve for point three: you can ensure that all of your users are covering their marginal costs, and then figure out the price point above that that maximizes revenue.

However, there are a few complications with this simplistic model. First, ChatGPT has a free tier which has marginal costs but which does not drive revenue. One reason to have this free tier is to be the top of the funnel for new users who might convert into subscribers; figuring out the lifetime value of these free subscribers — based on how many convert, how long they are subscribers, etc. — is a hard problem, but its one that any business with a freemium business model has to figure out. Another reason, however, that is very pertinent to ChatGPT, is the fact that there are plenty of alternatives. And, more often than not, these competitive pressures push the free tier to be overly generous relative to lifetime value.

And, of course, OpenAI is not a normal business: while I assume its revenue easily covers its marginal costs, it does not come close to covering its fixed costs; that difference is made up by investors, with the assumption that, at some point in the future, revenue will do exactly that, and kick out profit above that — just like Instagram.

How, though, will those revenue numbers be achieved? Well, there are two ways to increase revenue: increase the number of users, and increase the revenue-per-user.

Here you can definitely see the allure of enhancing the API: first, OpenAI is pushing new capabilities — like computer-control — down to the API sooner-rather-than-later; that should help get more developers on board. Second, OpenAI is increasing the stickiness of its API. Consider the difference I noted above between the Responses API and the Chat Completions API: the former, by being stateless, is trivial to switch out for another model that copied the API (which basically everyone has done). The former, by being stateful, will likely be trickier to switch out; moreover, that trickiness increases exponentially with every OpenAI-specific tool that it leverages. And, helpfully enough, the subscription business helps drive the development of those tools in the first place — OpenAI is its own first-best customer.

This approach doesn't just increase API users, and retain them more effectively, it also increases the likelihood of making more money off of them as well. First, to the extent OpenAI's API is better is the extent to which developers can build better products, which means they will have more success, which will drive more usage. Second, the more cross-usage OpenAI can drive of its API, the more usage it is driving overall, increasing revenue.

The Problem With Both
---------------------

The product side is a little trickier. ChatGPT almost certainly has very high brand awareness, although I'm sure there are still people trying it for the first time. A bigger problem is how many users only ever use the free tier; I suspect one of the reasons people were enamored with DeepSeek ( https://stratechery.com/2025/the-openai-critique-comparative-advantage-and-infrastructure-aggregation-theory-and-cheap-ai/?access_token=eyJhbGciOiJSUzI1NiIsImtpZCI6InN0cmF0ZWNoZXJ5LnBhc3Nwb3J0Lm9ubGluZSIsInR5cCI6IkpXVCJ9.eyJhdWQiOiJzdHJhdGVjaGVyeS5wYXNzcG9ydC5vbmxpbmUiLCJhenAiOiJIS0xjUzREd1Nod1AyWURLYmZQV00xIiwiZW50Ijp7InVyaSI6WyJodHRwczovL3N0cmF0ZWNoZXJ5LmNvbS8yMDI1L3RoZS1vcGVuYWktY3JpdGlxdWUtY29tcGFyYXRpdmUtYWR2YW50YWdlLWFuZC1pbmZyYXN0cnVjdHVyZS1hZ2dyZWdhdGlvbi10aGVvcnktYW5kLWNoZWFwLWFpLyJdfSwiZXhwIjoxNzQ0MzgwMDIyLCJpYXQiOjE3NDE3ODgwMjIsImlzcyI6Imh0dHBzOi8vYXBwLnBhc3Nwb3J0Lm9ubGluZS9vYXV0aCIsInNjb3BlIjoiZmVlZDpyZWFkIGFydGljbGU6cmVhZCBhc3NldDpyZWFkIGNhdGVnb3J5OnJlYWQgZW50aXRsZW1lbnRzIiwic3ViIjoiUnNldVlFOGpnSGVQcFdKZjFmQWI0QyIsInVzZSI6ImFjY2VzcyJ9.GnHDojYWn16IZAJTjKWOGYhlz4RZhb4c9K3z0XNsKbmqutgBnXk1z4ZCg1XiakgI2wexIAj69Nkk2L2Y4CmUeqA26Zm-i61Yi1ulIBRBlRj8IZwSwrO2KnnX7TP12PN0hehVtLa4pVBle9Rd-ew5DXyEOERNbe0Wx6jKFtlsfubYfLM34QmNk4tRk0VqIDQiJbEIFtZ7My3d4IX7kfFpIPKFPaAxw7dZRxUqzAywqOb6h4aLuvBNtvpmmsFDbUXuQEcYaWqmT3Wlb-pvhyUuq82P-otbKHdJ6IoIOFfUjNd8zoJXPNh29cLY6SrfN80wggNA7OTBFnPtzLJD1Sxteg ) is because the model on offering was better than the free offerings they were used to on ChatGPT. Third, anyone who is agentic enough to use ChatGPT enough to subscribe is probably more likely to be aware of alternatives, which helps put a cap on pricing. And now, thanks to this API release, some of the exclusive ChatGPT features are going to be available elsewhere.

In short, while its remarkable just how much money OpenAI is making from subscriptions — at least $25 million in revenue per month ( https://www.theinformation.com/briefings/openai-revenue-surged-from-200-a-month-chatgpt-subscriptions ) — there are some natural limits, both in terms of number of users and revenue-per-user, to just how big a subscription business can get, particularly a consumer-focused one.

There is, however, a workaround: advertising. Advertising simultaneously increases your addressable market, by making the product "free", and also uncaps your potential revenue-per-user, because it is advertisers footing the bill, not users. This is why Instagram is such a good business!

Again, though, it's not so easy for OpenAI. They have marginal costs to cover and zero advertising revenue to cover them; it's also not clear what kind of advertising will actually work with chatbots. It is in many respects safer and easier to, as Weil says, stick with being both a product and an API company, with business models that are broadly similar.

Those businesses, however, both have competition — particularly the API. That's going to be another natural limiter. On the other hand, a potential ad business (augmented with subscriptions, i.e. The Unified Content Business Model ( https://stratechery.com/2023/the-unified-content-business-model/?access_token=eyJhbGciOiJSUzI1NiIsImtpZCI6InN0cmF0ZWNoZXJ5LnBhc3Nwb3J0Lm9ubGluZSIsInR5cCI6IkpXVCJ9.eyJhdWQiOiJzdHJhdGVjaGVyeS5wYXNzcG9ydC5vbmxpbmUiLCJhenAiOiJIS0xjUzREd1Nod1AyWURLYmZQV00xIiwiZW50Ijp7InVyaSI6WyJodHRwczovL3N0cmF0ZWNoZXJ5LmNvbS8yMDIzL3RoZS11bmlmaWVkLWNvbnRlbnQtYnVzaW5lc3MtbW9kZWwvIl19LCJleHAiOjE3NDQzODAwMjIsImlhdCI6MTc0MTc4ODAyMiwiaXNzIjoiaHR0cHM6Ly9hcHAucGFzc3BvcnQub25saW5lL29hdXRoIiwic2NvcGUiOiJmZWVkOnJlYWQgYXJ0aWNsZTpyZWFkIGFzc2V0OnJlYWQgY2F0ZWdvcnk6cmVhZCBlbnRpdGxlbWVudHMiLCJzdWIiOiJSc2V1WUU4amdIZVBwV0pmMWZBYjRDIiwidXNlIjoiYWNjZXNzIn0.DR88JSCv3auckPUjG5EX7PSXtMofwQie6xRSNtcQlvexHIumymY-navkwaN00EvQHYHkkVarXTmKjuJ8iUFzi6ZVrz8RD0Dg6vKoseiuZh6mIoeeRWHqoUCcVnoRsE-9LklBMPYUOhFDFfrHHpwIRs8sKzEmvCfXwkBBdxQGGm5Xr2pyaHWZaQIglZlsby4t6EgUJ6eJuWYbwdmGwbxFXAKeCeZyRSJh0p5spNFdKIiwTTX0eMgBSd4c96dlGcGvbshsHJXXmDGDBw1XG4BnlIHuZ76SGALl4HNJflGZWfXTKBX-iQtFxlvRxVa1miZt9AkRoq1Gh9QSMa3WmZNHbQ ) ), will not have any competition at all! No one has — or likely will — reach the level of consumer penetration that ChatGPT has. Moreover, if they actually go in this direction, they can ensure they have no competition ever because they will be able to make more features free, making it impossible for competitors to economically match their offering. This is a hard path, to be sure, but I think it is the path that maximizes the chances of making the profits its investors are counting on.

In this view, the problem with the API business is the opportunity cost it imposes on building a consumer business. First, at least in the short term, there is the cost of running out of GPUs, which happened just last month ( https://www.tomshardware.com/tech-industry/artificial-intelligence/openai-has-run-out-of-gpus-says-sam-altman-gpt-4-5-rollout-delayed-due-to-lack-of-processing-power ) with the roll out of GPT 4.5. Second, there are all of the employees working on the API who aren't working on building the ad business (although, granted, a lot of them probably wouldn't want to). Third, APIs have hidden costs, like this bit from OpenAI's announcement:

*What this means for existing APIs*

* Chat Completions API ( https://platform.openai.com/docs/guides/text-generation ) : Chat Completions remains our most widely adopted API, and we’re fully committed to supporting it with new models and capabilities. Developers who don’t require built-in tools can confidently continue using Chat Completions. We’ll keep releasing new models to Chat Completions whenever their capabilities don’t depend on built-in tools or multiple model calls. However, the Responses API is a superset ( https://platform.openai.com/docs/guides/responses-vs-chat-completions ) of Chat Completions with the same great performance, so for new integrations, we recommend starting with the Responses API.
* Assistants API ( https://platform.openai.com/docs/assistants/overview ) : Based on developer feedback from the Assistants API beta, we’ve incorporated key improvements into the Responses API, making it more flexible, faster, and easier to use. We’re working to achieve full feature parity between the Assistants and the Responses API, including support for Assistant-like and Thread-like objects, and the Code Interpreter tool. Once this is complete, we plan to formally announce the deprecation of the Assistants API with a target sunset date in mid-2026. Upon deprecation, we will provide a clear migration guide from the Assistants API to the Responses API that allows developers to preserve all their data and migrate their applications. Until we formally announce the deprecation, we will continue delivering new models to the Assistants API. The Responses API represents the future direction for building agents on OpenAI.

To summarize:

* OpenAI has to spend resources supporting a deprecated API for at least a year.
* OpenAI has to spend resources supporting an API they no longer recommend using indefinitely. This means that the cost isn't just compute resources, but also significant engineering resources to update it with new models and capabilities.

This is the core issue: an API business is a customer support business; the Aggregator business is a consumer support business, where everyone else in the value chain serves you ( https://stratechery.com/2025/stargate-the-end-of-microsoft-and-openai/?access_token=eyJhbGciOiJSUzI1NiIsImtpZCI6InN0cmF0ZWNoZXJ5LnBhc3Nwb3J0Lm9ubGluZSIsInR5cCI6IkpXVCJ9.eyJhdWQiOiJzdHJhdGVjaGVyeS5wYXNzcG9ydC5vbmxpbmUiLCJhenAiOiJIS0xjUzREd1Nod1AyWURLYmZQV00xIiwiZW50Ijp7InVyaSI6WyJodHRwczovL3N0cmF0ZWNoZXJ5LmNvbS8yMDI1L3N0YXJnYXRlLXRoZS1lbmQtb2YtbWljcm9zb2Z0LWFuZC1vcGVuYWkvIl19LCJleHAiOjE3NDQzODAwMjIsImlhdCI6MTc0MTc4ODAyMiwiaXNzIjoiaHR0cHM6Ly9hcHAucGFzc3BvcnQub25saW5lL29hdXRoIiwic2NvcGUiOiJmZWVkOnJlYWQgYXJ0aWNsZTpyZWFkIGFzc2V0OnJlYWQgY2F0ZWdvcnk6cmVhZCBlbnRpdGxlbWVudHMiLCJzdWIiOiJSc2V1WUU4amdIZVBwV0pmMWZBYjRDIiwidXNlIjoiYWNjZXNzIn0.Wf0OdihSENWmyVTb9PqV8Qv4yLQE_ke_wPo1O9XmviZ-yy3WPN8j6KN_pZr1uC5U0K1gVLlqb41erOwSuTwxsLQ3WGmjH70cH1WMhGL9t328nWUQnSl4S0x2jZCq6eQjcXlAdttT9XB1_S1d9OhkvtlFSllvxNKK3tS35UyaCCFw7frjAq5LOKOH7dmbJDhNZR8cWs5GBwmH9_Z7gxIVvDtUtf6QebuY9IzS6HPnKA3FfWZcjZirnKzTxVOOIvUz4TyK_zK-LVpI9tPHjUgzpg9e8VkGvjafrEQYmahY6qGtb1BS00Wo8SyRyaucxAMCvykNhXk45CYYaWw067sZSw ).
